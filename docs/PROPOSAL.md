## 2. ที่มาและความสำคัญของปัญหา

ปัจจุบันการจัดการข้อมูลการเรียนการสอนในโรงเรียนไทยขนาดกลางยังกระจัดกระจายอยู่ในหลายเครื่องมือที่ไม่เชื่อมกัน:

- **คะแนนและเกรด** ถูกเก็บในไฟล์ Excel แยกของครูแต่ละคน ครูต้องคำนวณคะแนนรวมและเกรดเอง แล้วส่งออกไปรวมที่ฝ่ายวิชาการ เกิดความผิดพลาดและทำซ้ำหลายรอบ
- **การเช็คชื่อ** ใช้สมุดกระดาษหรือไฟล์แยก ทำให้สรุปสถิติการเข้าเรียนรายคนได้ยาก
- **การบ้านและการสื่อสาร** กระจายอยู่ในแอปภายนอก (เช่น LINE กลุ่มห้อง) ทำให้นักเรียนต้องพิมพ์ถามคะแนน/กำหนดส่งงานซ้ำ ๆ และครูตอบคำถามเดิมหลายครั้ง
- **การกำกับดูแล** ฝ่ายบริหาร/วิชาการ (Admin) ไม่มีร่องรอยการแก้ไขข้อมูล (audit trail) ที่ตรวจสอบย้อนหลังได้ เมื่อมีการแก้คะแนนหลังประกาศผลจึงตรวจสอบความถูกต้องได้ยาก
- **ความเป็นส่วนตัว** เครื่องมือทั่วไปไม่ได้ออกแบบให้นักเรียนเห็นเฉพาะข้อมูลของตนเอง สุ่มเสี่ยงต่อการเปิดเผยคะแนน/ข้อมูลส่วนบุคคลของผู้อื่น ซึ่งขัดต่อ พ.ร.บ.คุ้มครองข้อมูลส่วนบุคคล (PDPA)

ระบบเชิงพาณิชย์อย่าง Google Classroom ตอบโจทย์ฝั่งฟีด/การบ้านได้ดี แต่ไม่มีระบบคะแนน–เกรด–เช็คชื่อ–audit ที่ผูกกับโครงสร้างโรงเรียนไทย (ปีการศึกษา/เทอม/ห้อง/หน่วยกิต) ส่วน ERP โรงเรียนแบบเดิมมักเก่า ใช้งานยากบนมือถือ และไม่เน้นประสบการณ์ผู้ใช้

**โครงงานนี้จึงเสนอ** การพัฒนาระบบจัดการห้องเรียนแบบรวมศูนย์สำหรับโรงเรียนเดียว (single-tenant) ที่รวม การเช็คชื่อ คะแนน/เกรด การบ้าน และฟีดข่าวสาร ไว้ในระบบเดียว ภายใต้หลักการสำคัญคือ **"ครูเป็นผู้กรอกข้อมูล Admin เป็นผู้ตรวจสอบ"** พร้อมการควบคุมสิทธิ์การเข้าถึงอย่างเข้มงวด และร่องรอยการตรวจสอบครบทุกการแก้ไขข้อมูลของนักเรียน

## 3. วัตถุประสงค์ของโครงงาน

1. เพื่อพัฒนาระบบเว็บแอปพลิเคชันที่รวมการเช็คชื่อ การให้คะแนน/คำนวณเกรด การมอบหมาย/ส่งการบ้าน และฟีดข่าวสารของห้องเรียนไว้ในระบบเดียว
2. เพื่อออกแบบระบบควบคุมสิทธิ์ตามบทบาท (Role-Based Access Control) 3 บทบาท (Admin / Teacher / Student) ที่บังคับใช้กฎ "Authorization-first" ในทุกเส้นทางการเรียกใช้งาน
3. เพื่อสร้างระบบบันทึกร่องรอยการตรวจสอบ (Audit Log) ที่ครอบคลุมทุกการเปลี่ยนแปลงข้อมูลนักเรียน รวมถึงการบังคับระบุเหตุผลเมื่อแก้คะแนนหลังประกาศผล
4. เพื่อนำหลักการคุ้มครองข้อมูลส่วนบุคคล (PDPA) มาออกแบบในระบบจริง ผ่านการมองเห็นข้อมูลแบบจำกัด (L1 Visibility), การลบแบบ soft delete + anonymize และการให้ความยินยอม (consent)
5. เพื่อพัฒนาประสบการณ์ผู้ใช้ที่เหมาะกับแต่ละบทบาท โดยฝั่งนักเรียนเป็น mobile-first และเป็นไปตามมาตรฐานการเข้าถึง (WCAG 2.1 AA)

## 5. ประโยชน์ที่คาดว่าจะได้รับ

1. **ลดภาระครู** — กรอกคะแนนที่เดียว เห็นคะแนนรวม + เกรดทันที ไม่ต้องทำ Excel แยกหรือส่งออกหลายรอบ
2. **นักเรียนเข้าถึงข้อมูลตนเองได้ทันที** — คะแนน สถิติการเข้าเรียน กำหนดส่งงาน ลดการสอบถามซ้ำผ่าน LINE
3. **โรงเรียนตรวจสอบได้ตลอด** — ทุกการแก้ไขข้อมูลมี audit trail การแก้คะแนนหลังประกาศต้องมีเหตุผล
4. **คุ้มครองข้อมูลส่วนบุคคล** — ออกแบบตาม PDPA โดยให้นักเรียนเห็นเฉพาะข้อมูลของตน
5. **เป็นต้นแบบสถาปัตยกรรม** — ระบบจริงที่ใช้แนวคิด authorization-first, audit-by-design และ privacy-by-design ที่นำไปต่อยอดได้

## 6. ผู้ใช้งานเป้าหมายและบทบาท

ออกแบบสำหรับโรงเรียนไทยขนาดกลาง 1 โรงเรียนต่อระบบ (single-tenant) มีผู้ใช้ 3 บทบาทในบริบทต่างกัน

| บทบาท | จำนวนโดยประมาณ | อุปกรณ์หลัก | หน้าที่ | ข้อห้าม |
|-------|----------------|-------------|---------|---------|
| **Admin** | 1–3 คน | เดสก์ท็อป | ตรวจ audit/activity, นำเข้าผู้ใช้ตาม flow ที่มี, รีเซ็ตรหัสผ่าน, ดูภาพรวม, moderate | **เป็น read-only observer ของข้อมูลการสอน ไม่กรอกคะแนน/เช็คชื่อ/สร้างหรือแก้งานแทนครู** |
| **Teacher** | 30–80 คน | เดสก์ท็อป + แท็บเล็ต | สร้างวิชา, เช็คชื่อ, กรอกคะแนน, สร้าง/ตรวจการบ้าน, ประกาศ | ไม่เห็นห้องของครูคนอื่น |
| **Student** | 500–2,000 คน | มือถือ (95%+) | ดูคะแนน, ส่งงาน, ดูฟีด, ดูสถิติเข้าเรียน | ไม่เห็นข้อมูลของผู้อื่น |
| **ครูประจำชั้น** (คุณสมบัติเสริมของ Teacher) | — | — | เห็นภาพรวมห้องที่ดูแลทุกวิชา | — |

> หลักการออกแบบ "3 บทบาท 3 มุมมอง": Admin = เลนส์ตรวจสอบ (แน่น/มีเวลา-ไอดี), Teacher = เจ้าของพื้นที่ทำงาน (ควบคุม/แก้ไขได้), Student = บริโภคข้อมูล + ส่งงานง่าย (มือถือ/แตะง่าย)

## 7. ความต้องการของระบบ (Requirements)

### 7.1 Functional Requirements (สรุป)
- ผู้ใช้เข้าสู่ระบบตามบทบาท และเห็นเมนู/ข้อมูลตามสิทธิ์
- ครูจัดการวงจรชีวิตวิชา: สร้างวิชา → รับนักเรียน → เช็คชื่อ → กรอก/ประกาศคะแนน → มอบหมาย/ตรวจการบ้าน
- นักเรียน: สมัคร → เข้าห้องด้วย Class Code → ส่งงาน → ดูคะแนน/เกรด/ฟีด/แจ้งเตือน
- ระบบคำนวณคะแนนรวมจาก `fullScore` และเกรดรายวิชาอัตโนมัติ; Term GPA คำนวณเพื่อรายงานรอง ไม่ใช้แทนผลรายวิชา
- ครูเจ้าของวิชาดาวน์โหลด CSV สรุปคะแนนและการเข้าเรียนได้; นักเรียนพิมพ์/บันทึกผลการเรียนเป็น PDF ผ่านเบราว์เซอร์; Admin ดาวน์โหลด Audit CSV ตามสิทธิ์ผู้สังเกตการณ์
- ระบบบันทึก audit log ทุก mutation ที่กระทบข้อมูลนักเรียน

### 7.2 Non-Functional Requirements
- **ความปลอดภัย:** authorization-first ทุก route, เก็บรหัสผ่านแบบ bcrypt, ตรวจไฟล์ด้วย magic-byte, signed URL หมดอายุใน 5 นาที
- **ความเป็นส่วนตัว (PDPA):** L1 visibility, consent, soft delete + anonymize, สิทธิ์ของเจ้าของข้อมูล
- **ประสิทธิภาพ:** initial JS bundle < 250 KB gzipped, ฟีด server-paginate 20 รายการ, Lighthouse Performance ≥ 80 บน 3G
- **การเข้าถึง (Accessibility):** WCAG 2.1 AA, contrast ≥ 4.5:1, รองรับคีย์บอร์ดและ reduced-motion, ปุ่มแตะ ≥ 44px
- **ความน่าเชื่อถือของข้อมูล:** ทุก mutation อยู่ใน transaction (atomic), การลบใช้ soft delete เพื่อรักษาร่องรอย

### 9.2 แบบจำลองข้อมูลหลัก (Core Data Model)

ความสัมพันธ์หลักของระบบ:

```
AcademicYear ──< Term ──< CourseOffering >── Teacher
                              │   │
                Class ────────┘   ├──< Enrollment >── Student
                                  ├──< TimetableSlot ──< Session ──< AttendanceRecord
                                  ├──< ScoreItem ──< ScoreEntry
                                  ├──< Assignment ──< Submission ──< SubmissionVersion
                                  ├──< Material / Announcement
                                  └──< Comment
User (Admin|Teacher|Student) ──< AuditLog, Notification, FileAttachment, UserSession
```

**เอนทิตีสำคัญ:**
- **CourseOffering** — "พื้นที่ทำงานของครู" (workspace) เป็นศูนย์กลาง ทุกอย่าง (เช็คชื่อ/คะแนน/การบ้าน/ประกาศ/คอมเมนต์) อยู่ใต้เอนทิตีนี้ (แนวคิด workspace model, ADR-0012 — ไม่มี "Subject" กลางของโรงเรียน)
- **Enrollment** — ความสัมพันธ์ Student × CourseOffering ใช้ soft delete เพื่อรักษาร่องรอยคะแนน/การส่งงาน
- **AuditLog** — บันทึกเหตุการณ์แยกระดับ Critical / Important พร้อม snapshot ของเป้าหมายและเหตุผล
- **FileAttachment** — ตารางไฟล์แบบ polymorphic (owner_type + owner_id) ใช้กับทุกประเภทเจ้าของไฟล์

### 9.3 จุดเด่นเชิงสถาปัตยกรรม
- **Lazy session materialization** — ไม่มี cron ไม่ pre-generate คาบเรียน คาบจะถูกสร้างเมื่อครูเปิดเช็คชื่อจริง (ADR-0015)
- **Sparse attendance** — ไม่มีแถว = "ยังไม่เช็ค" ซึ่งต่างจาก "ขาด" อย่างชัดเจน (ADR-0016)
- **Sum-based scoring** — น้ำหนักของแต่ละช่องคะแนนกำหนดโดย `fullScore` โดยตรง (Σคะแนน/ΣfullScore × 100) ไม่มีช่อง weight แยก (ADR-0024)
- **Publish เป็นสัญญา (one-way)** — ประกาศผลแล้วย้อนกลับไม่ได้ การแก้หลังประกาศต้องมีเหตุผล (ADR-0018)
- **Architecture Decision Records (ADR)** — บันทึกการตัดสินใจเชิงออกแบบต่อเนื่องถึง ADR-0030 เป็นหลักฐานกระบวนการออกแบบ

## 10. ความปลอดภัยและความเป็นส่วนตัว (จุดเด่นของโครงงาน)

ระบบให้ความสำคัญกับความปลอดภัยและ PDPA เป็นหัวใจของการออกแบบ ไม่ใช่ของเพิ่มภายหลัง

- **Authorization-first** — ทุก API ตรวจบทบาท + ขอบเขต (ครูเป็นเจ้าของวิชานี้จริงหรือไม่) ไม่เชื่อ userId จากฝั่ง client
- **L1 Visibility** — นักเรียนเห็นเฉพาะข้อมูลของตน + ข้อมูลห้องระดับ metadata + รายชื่อสมาชิก (ไม่เห็นคะแนน/เลขประจำตัวของผู้อื่น)
- **Audit by design** — ทุกการเปลี่ยนแปลงข้อมูลนักเรียนถูกบันทึก แยกระดับความสำคัญ และห้าม log ความลับ (รหัสผ่าน/โทเคน/URL ที่เซ็น)
- **PDPA** — เก็บข้อมูลเท่าที่จำเป็น, ขอความยินยอม (consent + version), สิทธิ์ของเจ้าของข้อมูล, การลบใช้ soft delete + anonymize เพื่อไม่ทำลายคะแนน/การส่งงานของเทอมก่อน
- **ความปลอดภัยไฟล์** — ตรวจชนิดไฟล์จาก magic-byte (ไม่เชื่อนามสกุล), บล็อกไฟล์อันตราย (เช่น SVG ที่ฝังสคริปต์, executable), เสิร์ฟผ่าน signed URL ที่หมดอายุใน 5 นาทีเท่านั้น และสร้างหลังตรวจสิทธิ์แล้ว
- **การเก็บรหัสผ่าน** — bcrypt เท่านั้น (ห้าม plaintext/MD5/SHA1) · Admin รีเซ็ตได้แต่ระบบไม่เปิดเผยรหัสเดิม (ออกรหัสชั่วคราวแบบเห็นครั้งเดียว — ADR-0026)

## 11. วิธีดำเนินงาน (Methodology)

พัฒนาแบบ **incremental เป็นเฟส (phase-by-phase)** โดยแต่ละเฟสส่งมอบฟีเจอร์ที่ใช้งานได้จริงพร้อมชุดทดสอบ และยึดวินัย "1 commit = 1 concern" ผ่านการตรวจอัตโนมัติก่อน commit (typecheck/lint/test)

| เฟส | งานหลัก | สถานะ |
|-----|---------|-------|
| **Phase 0** | วางโครงโปรเจกต์ (scaffolding) | ✅ |
| **Phase 1** | ระบบยืนยันตัวตน + RBAC + นักเรียนสมัครเอง | ✅ |
| **Phase 2** | โครงสร้างวิชาการ + นำเข้าครู CSV + Class Code/QR/ลิงก์ | ✅ |
| **Phase 3** | โครงวิชา + การเป็นสมาชิกห้อง | ✅ |
| **Phase 4** | การเช็คชื่อ | ✅ |
| **Phase 5** | ระบบคะแนน + สรุปผลรายเทอม | ✅ |
| **Phase 6** | การบ้าน + การส่งงาน + คอมเมนต์ + อัปโหลดไฟล์ | ✅ |
| **Phase 7** | ฟีด + การแจ้งเตือน | ✅ |
| **Phase 8** | เครื่องมือตรวจสอบของ Admin (audit) | ✅ |
| **Phase 9** | ปรับ UI/UX + 3D + hardening ก่อนเปิดใช้ | 🔄 |

> หมายเหตุ: เครื่องหมาย ✅ หมายถึง core implementation ของเฟสนั้นมีแล้ว ไม่ได้หมายความว่า acceptance/completeness audit ปิดทุกข้อ ระบบ deploy บน Vercel + Neon + private Cloudflare R2 แล้ว และต้องผ่าน Release A เรื่องเอกสาร รายงาน/ส่งออก QA และ functional completeness ก่อนเริ่ม feature domain ขนาดใหญ่

## 13. การทดสอบและการประเมินผล

ใช้กลยุทธ์ทดสอบแบบพีระมิด:
- **Unit tests (Vitest)** — โมดูลคำนวณคะแนน/เกรด/GPA, การ validate, การ generate รหัสชั่วคราว (เป็น pure functions)
- **Integration tests** — เน้น **เมทริกซ์สิทธิ์ (permission matrix)** เป็นด่าน CI (ทุกกฎสิทธิ์ต้องมีเทสต์) และเส้นทาง mutation สำคัญ
- **E2E (Playwright)** — เส้นทางวิกฤต เช่น สมัคร→เข้าห้อง→ส่งงาน, ครูกรอก/ประกาศคะแนน, นักเรียนเห็นคะแนนหลังประกาศ
- **Security tests** — ตรวจ static + ทดสอบการเข้าถึงข้ามสิทธิ์ (เช่น นักเรียนพยายามดูคะแนนผู้อื่น ต้องได้ 403)
- **CI Pipeline (GitHub Actions)** — lint, typecheck, test, build ต้องผ่านก่อน merge เข้า main (ผ่าน Pull Request เท่านั้น)

เกณฑ์ประเมินความสำเร็จ: ฟีเจอร์ตามขอบเขตทำงานครบ, ชุดทดสอบสิทธิ์ผ่านทั้งหมด, ผ่านมาตรฐานการเข้าถึง WCAG 2.1 AA, และ Lighthouse Performance ≥ 80
